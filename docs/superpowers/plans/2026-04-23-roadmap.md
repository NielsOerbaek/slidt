# Slidt implementation roadmap

**Status:** Living document — plans get written subsystem-by-subsystem as each previous one lands.
**Spec:** `docs/superpowers/specs/2026-04-23-slide-webservice-design.md`

Six independent subsystems. Each one produces working, testable software on its own. Later subsystems build on earlier ones. Natural order is 1 → 2 → 3 → 4 → 5 → 6, with small inversions possible (4 can come before 3; 5 can start once 3's agent-chat shell exists).

## Subsystems

### 1. Core renderer + template system
**Delivers:** Pure-TypeScript `render(deck, theme, templates) → HTML` library that runs in both Node and browser. Handlebars helpers (`fmt`, `eq`, `default`), field schema validation, CSS auto-scoping, 13 seeded built-in SlideTypes ported from `generate.py`, ANTAL-Theta default theme extracted from `styles.css`, golden-HTML tests.
**Depends on:** nothing.
**Plan:** `2026-04-23-01-core-renderer.md` (written).
**Rough task count:** ~15.

### 2. Data model + API + auth
**Delivers:** Postgres schema via Drizzle, SvelteKit API routes for CRUD on decks, slides, themes, templates, and assets. Email/password auth with argon2id and HTTP-only session cookies. `create-user` and `reset-password` CLIs. Local-disk asset storage under `./data/assets`. Integration tests against a real Postgres container. Verifiable end-to-end with `curl`.
**Depends on:** Plan 1 (imports the renderer types and validation).
**Plan:** `2026-04-23-02-data-api-auth.md` (written).
**Rough task count:** ~20.

### 3. Editor UI
**Delivers:** SvelteKit pages for `/decks`, `/decks/[id]`, `/themes[/:id]`, `/templates[/:id]`, `/share/[token]`. Form editor driven by `SlideType.fields`, live preview using the Plan 1 renderer, slide list with drag-reorder. Agent chat panel shell (no LLM wired yet). Autosave on form blur.
**Depends on:** Plans 1 and 2.
**Plan:** `2026-04-23-03-editor-ui.md` (written).
**Rough task count:** ~15.

### 4. PDF export
**Delivers:** Playwright-backed PDF rendering endpoint (`GET /decks/:id/export.pdf`), appendix stitching via `pdf-lib`, font preloading for Inter + Neureal, visual regression tests against baseline PNGs for each built-in SlideType.
**Depends on:** Plan 2 (for the deck API). Can ship before or after Plan 3.
**Plan:** to be written after Plan 2 ships.
**Rough task count:** ~8.

### 5. Agent
**Delivers:** Claude API integration (Sonnet 4.6 for content/theme, Opus 4.7 for new slide types). Streaming via Server-Sent Events. Tool handlers: `list_slides`, `get_slide`, `patch_slide`, `add_slide`, `delete_slide`, `reorder_slides`, `update_theme`, `create_slide_type`, `list_slide_types`. Handlebars-AST + CSS-safe-rule guardrails. Agent chat UI wired up in the editor. Per-tool-call undo via inverse patches.
**Depends on:** Plans 1 (renderer), 3 (agent chat shell). Plan 4 not required.
**Plan:** to be written after Plans 3 ships.
**Rough task count:** ~15.

### 6. Migration + deployment
**Delivers:** `import-deck` CLI that maps the existing ANTAL-Theta `slides/` directory into the DB, copies assets to the data volume, extracts the theme from `styles.css`, and diffs the rendered PDF against `ANTAL-Theta-slides.pdf`. Docker Compose stack with `app`, `postgres`, `caddy`. Backup script (nightly tarball of `./data/` to OneDrive). `/healthz` endpoint. README with deploy instructions.
**Depends on:** all previous.
**Plan:** to be written last.
**Rough task count:** ~10.

## Why sequential

Each subsystem's plan is an executable document with exact file paths, complete code for every step, and a TDD cycle per task. Writing later plans before the earlier ones land would mean inventing file paths, type signatures, and test helpers for code that doesn't exist — those details shift once reality hits. Writing them sequentially means each plan is informed by the real code from previous subsystems.

The roadmap itself is cheap to write and gives us shared scope visibility without committing to speculative detail.
